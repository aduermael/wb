# Release

This is maintainer documentation for publishing `wb` releases.

Releases are intentionally built from a local Mac for now. There is no GitHub Action because the release binaries need to be built on macOS 26+.

## Requirements

Build requirements:

- A Mac with macOS 26.0 or newer
- Xcode 26 or newer, with Swift 6.2 SDKs

Publish requirements:

- `gh` authenticated to `aduermael/wb`
- `gh` access to `aduermael/homebrew-tap`
- for npm, `npm` authenticated to an account that can publish `@aduermael_/wb`
- for `--publish-only`, the release assets generated by `./release.sh <version> --build-only`
- for `--tap-only`, a published GitHub Release with both macOS tarballs or checksum assets
- the same clean checkout and commit used for the build, except for `--npm-only`

## All In One

Use this only on a Mac that also has `gh` and `npm` authenticated:

```bash
./release.sh 0.1.0
```

The script builds `arm64` and `x86_64` release binaries, packages tarballs, writes SHA-256 checksums, pushes `main`, creates/pushes an annotated tag like `v0.1.0`, and creates or updates the GitHub Release.
It then updates the Homebrew tap and publishes the npm package.

Release binaries are signed before packaging. By default, signing uses ad-hoc codesigning (`codesign -s -`), so an Apple Developer ID certificate is not required. Set `WB_CODESIGN_IDENTITY` to use another local signing identity, or set `WB_CODESIGN=off` to skip signing.

Migration note: old sessions created in a cwd-local `wb/` directory or a non-root `.wb` directory are not moved automatically. Use `WB_DIR` to point at the old directory when needed, or manually migrate its contents into the new git-root `.wb`.

Do not ship or track `.wb/environment.json` in sample projects. That file is public metadata, but it intentionally defines the WebKit website data profile for a local environment.

It also updates the Homebrew formula in `aduermael/homebrew-tap` when both macOS release assets are present. The tap update uses local checksum files when available and falls back to the published GitHub Release assets, so a missing local `dist/` checksum should not leave the tap stale. To publish without touching the tap:

```bash
./release.sh 0.1.0 --no-tap
```

To publish without touching npm:

```bash
./release.sh 0.1.0 --no-npm
```

Existing tags are reused when they already point at `HEAD`. If a tag exists but
needs to be moved intentionally, use:

```bash
./release.sh 0.1.0 --publish-only --force-tag
```

To use another tap repository:

```bash
./release.sh 0.1.0 --tap-repo owner/homebrew-tap
```

To update only the Homebrew tap for an already-published release:

```bash
./release.sh 0.1.0 --tap-only
```

To publish only the npm package for an already-published release:

```bash
./release.sh 0.1.0 --npm-only
```

## Split Build And Publish

If the Mac does not have `gh`, build on the Mac:

```bash
./release.sh 0.1.0 --build-only
```

That creates:

```text
dist/v0.1.0/wb-macos-arm64.tar.gz
dist/v0.1.0/wb-macos-arm64.tar.gz.sha256
dist/v0.1.0/wb-macos-x86_64.tar.gz
dist/v0.1.0/wb-macos-x86_64.tar.gz.sha256
```

Copy `dist/v0.1.0/` to an environment with `gh` authenticated to `aduermael/wb`, using the same checkout and commit, then publish:

```bash
./release.sh 0.1.0 --publish-only
```

`--publish-only` publishes or updates the GitHub Release, updates the Homebrew
tap, and publishes the npm package. Use `--no-tap` or `--no-npm` to skip either
registry.

To release only one architecture:

```bash
./release.sh 0.1.0 --build-only arm64
./release.sh 0.1.0 --publish-only --no-npm arm64
```

The npm package supports both `arm64` and `x64`, so npm publication requires
both macOS release assets to exist.

## npm Package

The npm package is a thin Node wrapper around the GitHub Release binary. The
normal release path publishes it automatically after the matching GitHub Release
exists with both `wb-macos-arm64` and `wb-macos-x86_64` tarballs plus their
`.sha256` checksum files.

To publish npm by itself:

```bash
./release.sh 0.1.0 --npm-only
```

`--npm-only` does not create, move, or push git tags, and it does not require a
clean worktree. It requires the matching GitHub Release assets to already exist.

To inspect the generated package manually, prepare a versioned package
directory:

```bash
scripts/prepare-npm-package.sh 0.1.0
```

Then inspect the package contents:

```bash
npm pack --dry-run dist/v0.1.0/npm
```

The source `package.json` intentionally stays at `0.0.0-development`. The
staging script writes the release version into the generated package so
`postinstall` downloads the matching GitHub Release tag.
